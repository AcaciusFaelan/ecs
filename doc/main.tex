\section{Funktionsweise}
\subsection{ECS}
\begin{wrapfigure}{r}{0.4\textwidth}
	\centering
	\includegraphics[width=0.38\textwidth]{assets/ECS_Simple_Layout.png}
	\caption{Simples ECS Layout}
\end{wrapfigure}

Ein ECS ist ein Software-Architekturmuster.
Es wird häufig in der Videospiel-Entwicklung vorgefunden um Spielwelt Objekte darzustellen.
Ein ECS besteht meistens aus Entitäten, die aus Datenkomponenten zusammengesetzt sind, als auch Systemen, die auf diesen Komponenten operieren.
Jede Entität ist nicht durch eine Typen-Hierarchie definiert, sondern durch die, mit diese Entität, verbundenen Komponenten.
Systeme führen ihre Logiken global über alle Entitäten mit den benötigten Komponenten aus.
Das verhalten von Entitäten kann dynamisch während das Programm oder Spiel läuft angepasst werden.
Dies passiert durch Systeme die Komponenten zu den Entitäten hinzufügen, entfernen oder modifizieren.

\subsubsection{Entitäten}
Eine Entität repräsentiert ein Allzweck Objekt.
Das kann sein ein Tier, eine Pflanze und der Spieler Charakter.
Eine Entität besteht aus einer UID oder Unique Identifier.
Das ist meistens ein Integer Type.

\subsubsection{Komponenten}
Eine Komponente charakterisiert eine Entität indem sie einen bestimmten Aspekt trägt.
Dafür hält sie die Daten um diesen Aspekt zu modellieren.
Zum Beispiel, jedes Objekt, in einer Spielwelt, das Schaden nehmen kann, hat vielleicht eine Lebenskomponente verbunden mit seiner Entität.

\subsubsection{Systeme}
Ein System ist ein Prozess oder Funktion die über alle Entitäten mit den gewünschten Komponenten agiert.
Zum Beispiel, ein Physiksystem fragt vielleicht nach Entitäten die Masse-, Beschleunigungs- und Positionskomponenten haben.
Dann geht es über das Ergebnis und macht seine Physik-Berechnungen für jede Entität mit dem Set an Komponenten.

\subsection{OOP}
Bei OOP gibt es mehrere Möglichkeiten Spielobjekte darzustellen.

	\item Komposition
\end{itemize}

\subsubsection{Vererbung}
Vererbung ist ein Mechanismus.
Dieser erlaubt es einem, ein Objekt oder eine Klasse auf einer anderen aufzubauen.
Dadurch kann die „Chield-Class“ Funktion der „Parent-Class“ nutzen ohne sie erneut implementieren zu müssen.

\subsubsection{The diamond problem}
Das Diamantproblem tritt nur bei Vererbung auf.
Dieses Problem tritt auf, wenn eine Klasse A an eine Klasse B und C vererbt.
Die Klasse D erbt dann von beiden B und C.
Klasse A definiert eine Funktion die von B und C überschrieben wird.
D überschreibt diese Funktion nicht.
Nun wird die Funktion bei D aufgerufen.
Welche Funktion soll nun aufgerufen werden?
Beide B und C haben diese Funktion überschrieben.
Das Problem wird so genannt, da die Form des Klassen vererbungs Diagramms die Form eines Diamanten hat.


\subsubsection{Komposition}
Komposition ist ein oft genutztes Designmuster.
Es versucht Wiederverwendung von Code zu erreichen ohne Vererbung zu benötigen.
Komposition simuliert Vererbung indem die „Chield-Class“ eine Kopie von einer „pseudo-parent“ Klasse als Feld einbindet.
Dieses Implementiert die gemeinsame Funktionalität der beiden Klassen.
Anschließend werden Funktionen, die normalerweise auf der „Chield-Class“ aufgerufen werden worden, auf der „Parent-Class“ aufgerufen.
Dies nennt sich method forwarding.

\section{Implementierung}
\subsection{Entity/Komponenten Manager}
Alle Klassen und Funktionen sind in einem \texttt{namespace} namens \texttt{ecs} definiert.

\subsubsection{definitions.hpp}
Hier wird der \texttt{Entity}-Typ definiert.
Hier wird außerdem der Wert \texttt{NULL\_ENTITY} definiert, was der höchst mögliche Wert eines \texttt{uint32\_t} ist.
Dieser Wert ist $2^{32}-1$.

\subsubsection{sparse\_set.hpp}
Hier wird die \texttt{SparseSet}-Klasse definiert.
Das \texttt{SparseSet} ist dafür zuständig, genau einen Komponenten Pool zu speichern.
Es speichert genau drei \texttt{Vektoren}.
Einen für die Komponenten namens \texttt{dense}.
Einen für \texttt{Entity} zu \texttt{Komponenten-Index} namens \texttt{sparse}.
Und einen für \texttt{Komponenten-Index} zu \texttt{Entity} namens \texttt{indices}.
Die Klasse besitzt außerdem noch Funktionen um \texttt{Komponenten} hinzuzufügen, sie zu entfernen, sie anzufordern und um abzufragen ob eine \texttt{Entity} sie besitzt.

\subsubsection{Registry.hpp}
Hier wird die \texttt{Registry} definiert.
Sie ist dafür zuständig die \texttt{Entities} und die \texttt{SparseSets} zu managen.
Sie stellt ein Interface bereit um mit den \texttt{SparseSets} zu interagieren.
Sie speichert für jeden \texttt{Komponenten}-Typ ein \texttt{SparseSet}.
Die \texttt{Registry} erstellt immer nur dann eine \texttt{SparseSet}, wenn der \texttt{Komponenten}-Typ noch nicht vorhanden ist.
Diese Klasse besitzt, wie auch die \texttt{SparseSet}-Klasse, Funktionen zum hinzufügen, entfernen, anfordern und abfragen von \texttt{Komponenten}.
Darüberhinaus besitzt sie noch Funktionen um \texttt{Entities} zu erstellen und um \texttt{Views} zu erstelle.

\subsubsection{view.hpp}
Hier wird die \texttt{View}-Klasse definiert.
Sie ist nur interesant für \texttt{Systems}.
Die \texttt{View}-Klasse nimmt sich ein oder mehrere gewünschte \texttt{SparseSets} von der \texttt{Registry} und fürt eine, als Funktionsargument verpackte, Funktion für die \texttt{SparseSets} aus.
Diese Klasse wird nicht manuel konstruiert.
Die \texttt{Registry} stellt dafür eine Funktion bereit.

\subsection{Komponenten}
\texttt{Komponenten} sind einfache C++ \texttt{Strukturen}.
Diese können alle möglichen Felder beinhalten.

\subsection{Systeme}
\texttt{Systeme} sind einfache C++ \texttt{Funktionen}.
Diese nehmen als Argument einen \texttt{Reference} zur \texttt{Registry} auf der sie operieren sollen.
Oder sie können auch einfach die Argumente (\texttt{Entity, Comp1, Comp2, ...}) nehmen.
Zurück geben diese \texttt{Funktionen} nichts, da sie ja nur mit den \texttt{Komponenten} operationen anstellen.
